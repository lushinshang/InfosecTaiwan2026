## 內容

**主持人（ISC2 台北分會理事長）:** 大家早安，歡迎大家來參加我們這個 ISC2 台北分會的分場。我先自我介紹一下，我是（ISC2 台北分會）理事長。今天我們很開心邀請到 ISC2 的理事林家瑋，來幫我們介紹這一場「DDoS 只有爆流量？從 OSI（七層）拆解 AWS DDoS 的防護」。我們的（Ray）他在這個領域是 AWS 系統的專家，他本身是全球首批的 AWS Hero，而且同時也是大中華區的第一位，然後還有很多很多的頭銜——我剛剛在看手機，努力記起來，等一下我年紀有點大，那邊記不住。我就把時間交給他，熱烈的掌聲歡迎 Ray Lin。

**林家瑋（Ray Lin）:** 好 OK 謝謝我們理事長的介紹。對於今天早上的議程有興趣的（各位），歡迎大家早安。就是我們的場（次）就是要比主（議程）多，這就是我們今年暫定的目標——那看起來應該有了。

好，那今天分享的議題是講 DDoS。因為昨天有朋友說他想要來聽我講，然後我說我會講到 AWS，然後他說「我對 AWS 沒興趣」，我說沒關係，因為我 20 分鐘不是講 AWS，20 分鐘在講 DDoS，所以他就來了。OK，那我今天代表在 ISC2 台北的一個分享。其實我也在很多的國際組織跟社群，在座應該有很多（人認識我）。好，我就先快速地自我介紹，因為今天時間比較緊，所以我就不花太多時間介紹了。我大概在（這個）團體才在做 20 年以上的時間，然後著重的領域就四大領域：從資訊安全、到雲端、到 AI、到專案管理。因為我自己本身在八年前就開始做 AI，所以當時開始去學微軟的 AI，取了一個微軟 AI 專家以後，再來這幾年拿了 AWS 的 AI 專家跟雲端安全聯盟的 AI 專家。其他比如說我也有取得像 CISSP，在座也有很多 CISSP，目前全球資安界的黃金證照。我都說它是資安界的 PMP，但是數量還沒有 PMP 多，還有五倍（差距）。

我目前在兩間公司任職，分別做資安長跟數位長，另外也有在一些培訓單位授課，譬如說也許有我的 PMP 的學員，還有 AWS 的學員，以及我有教授 ISC2 的證照，但是我目前比較少教，好因為我最崇拜的人都在教 ISC2 的證照，所以我要去聽他們的課，我們要講。好那其他部分我在一些國際組織就是有任(何職位)，包括像 ISC2 台北、PMI 台灣跟 TWCSA，就是今天一起主辦這個 InfoSec 的單位，然後以及我自己有成立一些社群。好那其他就請參閱。

OK，那我們就直接進入今天的主題。好，那我們今天主題講的是 DDoS，但是其實我們不能只知道 DDoS。DDoS 叫做分散式阻斷攻擊，所以它多一個 D，多一個 D 代表它是分散式，它是從很多的地方、很多的 IP，它是分散的攻擊。那我們在講 DDoS 之前要先講 DoS，它叫阻斷服務，只要能夠讓你們的服務中斷，都叫 DoS、都叫阻斷服務。所以在講 DDoS 之前，我要先講什麼叫阻斷服務。

好，那我舉幾個比較舊的攻擊案例。不知道大家有沒有聽過碎片攻擊？我們今天在做網路傳輸的時候，譬如說傳輸檔案、譬如傳一個 1TB 的檔案，你不可能是整個 1TB 直接傳，它一定要把它切成很多段。切成很多段以後，對方收到你這很多段的小檔案，他要把它組合成一個，他才能用。所謂的碎片攻擊是指說：我今天在丟很多段的碎片的時候，它要重組、要告訴你的順序，就是它現在是第幾號到第幾號、下一個是第幾號到第幾號，然後它才能重組成一個完整的 1TB 的檔案。可是如果是一個惡意攻擊者，他可能不是給你一個正常的順序。好這邊舉一個例，假設有三個碎片，一二三，他會告訴你說我這個碎片的 offset 是多少、offset 是多少、offset 是多少，我這樣才知道說它要怎麼組。但是他第三個碎片，你會發現說它跟前面一個碎片的範圍有重疊——因為它（第一個）是 0 到 100，但是第三個碎片它是從 80 開始，那是跟前面的 0 到 100 有重疊。那重疊的時候它重組就會失敗、就會出錯。甚至說我今天丟了很多碎片給你，然後我沒有（丟完）、我可能只丟一半，那你就一直等、一直等我丟，這個時候你就會消耗一些資源一直在等。所以這個叫碎片攻擊，這個攻擊也蠻久了，基本上現在很多的作業系統都已經修復這個問題了，但是它還是有機會發生，尤其是在一些舊的系統。比較常見的攻擊名稱叫淚滴攻擊。

再來第二個 DoS，也是比較（舊）的，它叫死亡之 Ping（Ping of Death）。如果有唸過現在的（網路課程），就覺得我在講——什麼叫死亡之 Ping？一般來說我們要測網路有沒有通，我是（用）Ping 一下，一個 8.8.8.8，它如果 Ping 通，它會回傳我資料。基本上它有一個封包上限、大小上限就是 64K。如果我們現在 Ping 的時候，Ping 它的這個大小——因為 Ping 它其實還是可以去做切、切這個碎片、切分段，最後它接收以後把它組合起來，是超過 64K、超過它的上限的話，它可能也會有問題、就會出錯，舊的系統它可能會當機。所以這個讓它當機、不能提供我們網站服務，這都叫 DoS。

那我們今天主軸要講的是分散式阻斷攻擊、相關的 DDoS 攻擊。我們就會從 OSI 七層來去看所謂的 DDoS 攻擊有哪些常見的。我們知道這些常見的攻擊以後，我們就會發現說其實 DDoS 攻擊很難防禦，甚至是無法防禦，最後的解決措施要嘛就是找 ISP 業者、比如說中華電信這種去洗流量，要嘛就是緩解——所謂緩解就是它沒有辦法百分之百防禦，它只能降低衝擊這種感覺。如果有認真研究 DDoS，你會有這種感覺。

我們來看一下。這邊除了 DDoS，其實還有摻雜一些 DoS，就是它不一定是大流量、它可能也不一定是分散式的，我就會綜合去講解。那 OSI 七層在座的各位應該大部分都知道，如果不知道的我也沒有辦法現場教學，因為它這個算是網路蠻基本的。所以我們從第一層實體層、第二層資料鏈結、第三層網路層、第四層傳輸層、第五層會話層 Session、第六層呈現層、第七層 Application 應用層。這七層裡面只有四層會發生 DDoS 跟 DoS 的攻擊，只有四層，就是用（投影片上）標的、比較偏白色的這個。右邊講的這些攻擊我都會講，結果等下短短的 20 分鐘我會把這些攻擊都講完。

那我們 DDoS 攻擊，它的第一個 D 叫分散式，所以它也是阻斷，只是它透過分散式的大流量來讓你的服務被塞滿。塞滿什麼？塞滿你的網路頻寬、塞滿你的 CPU 負載、塞滿你的記憶體、塞滿你的連線數，這個都叫 DDoS 攻擊。

因為我要講 DDoS 攻擊，那還是要講一個實際案例，就是我真實有發生的。我自己有一個公司官網，公司官網裡面都會有留言板，就大家可以去提問或者是跟我聯繫。我用的是免費的人家的留言板套件，它是有每個月的免費發送留言的次數。然後就有一個很瞎（的事）：它就用不知道什麼機器人去我那個留言板，可能 key 了很多惡意指令，然後一直 key 一直 key 一直發送，所以我就在 5 分鐘內瞬間收到 200 多封的 E-mail——因為它留言我就會刷 E-mail。所以你看，我這邊沒有列時間、只有列日期，基本上這個就是幾分鐘就發了幾百封，然後 DDoS 攻擊。最後我這個留言板超過留言上限，我就不能收 E-mail 了，所以假設這一個月有人又要發這個留言來跟我聯繫，我就收不到，因為打到上限了。

當時我的官網，我繼續看它的一個（負載）——這個不是 CPU，這個應該是網路流量的負載，在某一個時間點瞬間變高。很剛好我大概這兩個月就遇到四次，兩個月就被攻擊四次，一樣的手法。不過它 IP 都封鎖，還是一樣，它還是可以透過 Pitcher IP 來攻擊。剛好都是我的晚上、下班時間，然後我一看到被攻擊了，趕快開電腦去把它封鎖掉。

再來，因為我在檢查這個（流量）log 記錄的時候有一些資訊可以看，所以我在 AWS 上有建了一個 Dashboard——譬如說也許大家有的是第一次看到類似的 Dashboard——我會去收集它的 log 去統計說，哪一個 IP 在這段時間請求了多少次。理論上我這個叫官網、官網一個月沒有幾個人去看，但是你在幾個小時之內可以發送幾千個請求跟幾萬個請求，看起來就很奇怪，雖然它不一定被封鎖，但是就很怪。再來，我去看了一下，有一些是台灣來的請求，有些是來自於美國的——我的網站是中文的，為什麼會有美國人來看？應該不太會有。再來我看到我的 CPU 負載在這一瞬間超過了 40%，我的 server 一到 40% 它就 down 了，沒有到 100% 就 down、40% 就 down 了，然後我就要重開機。然後網路流量也一樣有一些尖峰。

後來針對類似的攻擊，最簡單的方式就是加黑名單，就是針對這些 IP 就做黑名單。但是如果每次被 DDoS 攻擊你都要手動加黑名單是不是有點笨？因為它有時候是分散式攻擊，它就是很多個 IP，那你要加幾個 IP、你隨時都要待在那。所以我後面有設置自動化——所謂自動化是說，它這些請求只要 5 分鐘內超過 2000 次，我就直接自動把它加入黑名單，然後就永久封鎖。因為我的網站客戶也不多，所以封鎖錯了也沒差，不缺客戶，（真的有需要）要就直接聯繫我。所以我就設了一個機制，5 分鐘超過 2000 次請求就自動加黑名單，然後它就永久被我封鎖掉了。所以我大概經歷了幾次以後，手動封鎖完，我後面都用自動化。

那我們先來講這幾層的 DDoS 攻擊跟 DoS 攻擊。我們講網路層跟傳輸層，這兩層我會一起講，是因為有很多的單位他們也是一起講——第三層、第四層一起講。為什麼會一起講？主要是因為有些攻擊它可以是第三層也可以是第四層，它兩層都有關係，所以一般來說它是兩層一起講比較多。我等一下講的時候，大家可以去思考一下它可能是在第三層還是第四層。第三層我簡單講一下，大概就偏向我們的 IPv4、IPv6，還有譬如說 ICMP 的這些協定，比如說 Ping、Traceroute，然後以及像是定址還有路由，路由也算是第三層。如果是傳輸層第四層的話，比較偏向是 TCP 跟 UDP 的協定，就是我們的可靠傳輸跟不可靠的傳輸這兩種。

那我們來看一下這些攻擊它可能是哪一層。第一個來講 ICMP 的 Flood，Flood 就是洪水，跟洪水一樣，我用 ICMP 的這些協定包用洪水攻擊。ICMP 有什麼比較知名的指令？就是 Ping。我瘋狂地用很多台主機去 Ping 你的主機，那你就會收到很多小封包。Ping 的話它最大的封包就是 64K，所以你一直 Ping，你 Ping 多少它就會回傳多少，這是 1 比 1。所以一般來說 ICMP 的 Flood 你一定要很多台你才能造成對方負擔，不然的話你是 1 比 1，那你就要準備很多台去攻擊它。

那 Ping 的話，我們除了去 Ping 一般 Internet，我們也可以 Ping 內網，可以 Ping 內網的這些 192.168 的這些內網 IP。一般來說我們內網在處理這些 Ping 的時候，主要是要找到它的 MAC 位置，就是它的網卡的實體位置。所以一般來說你在找主機的時候，都會去（用）廣播的方式去問說「這個 IP 是誰」，就是你要去 ARP、去問最後一個 IP，確認說現在這個 IP 位置是哪一個 MAC 位置。所以你發送一個 ARP 以後，它是會問所有在內網、有 IP 的主機，然後它們都會回傳你說「是不是我、是不是我」。假設我今天偽造別人 IP，譬如說偽造你同學、你的 IP，然後去做這個廣播，那其他機器就會回傳你資訊——我只發了一則，255 台都會回它，我再發一則，255 台都會回它，有沒有變多的感覺？這個就叫放大攻擊。所以最有名的就是這個 Smurf（攻擊），就是我偽造別人的 IP，然後我去問廣播，然後廣播就會把大家的回應回給被我偽造的人，就是受害者，這個都是放大攻擊。我們會發現後面很多的阻斷跟 DDoS，它很多都是放大攻擊，就是我只要發一個 byte，你會收到幾十 byte、幾百 byte，就是它放大幾倍、幾十倍。

下一個，我先問一下好了，這個 ICMP Flood 跟 Smurf，大家覺得是第三層還是第四層的攻擊？（聽眾：第三層。）第三層，因為它用的是 ICMP。

再來第二個攻擊，這個叫 UDP 的 Reflection Attack。UDP 一般來說我們可能會發送 UDP-123 然後 UDP-53，這兩個一個是（時間校時）——就是（NTP 時間），然後一個是 DNS 的查詢，用 UDP 的方式。一般來說如果以 DNS 為例，就是我發 UDP-53，假設我請求一個 60 byte、就是我發送的通報是 60 byte，那一般來說你會收到大概 3000 byte，這也是放大 50 倍。如果我又偽造別人的 IP、偽造你的 IP，然後我去發 UDP-53，然後它就會傳你 50 倍，所以我發一次你就 50 倍，50 倍的攻擊，我就可以把它的流量衝爆。

那更厲害的是這個，大家有聽過這個叫 NTP MonList 的指令嗎？NTP 就是我們發送 123 port，這個指令是用來做什麼？就是你詢問它以後，它會回傳你最近 600 個有來問的 IP，就是它回覆你最近 600 個有來詢問的人。所以這個叫放大幾百倍——就是最近 600 個。所以這邊有一個數字，它可以放大 206 倍，因為我發那個 NTP MonList，它會回傳給被我偽造的人 206 倍，這個東西很瞎。目前 NTP Server 大部分都已經把這個指令給關掉了，但是還是有一些舊的還是有。所以它這個都是放大東西。那請問 UDP 的這種 Reflection 的攻擊（是哪一層）？（聽眾：第四層。）第四層，因為它是用 UDP，用 TCP、UDP 大部分都會跟第四層有關係。

好，再來看一下下一個，這個叫 Connection 的 Flood、連接的洪水。一般來說我們的 Server 要跟我們客戶端先建立連線才會傳資料，所以我會建一個 Connection。講這個建立 Connection，大家知道 TCP 的三段交握嗎？我來快速講一下三次交握，這個簡報如果有加社群的我都會丟在 LINE 群，所以大家不一定要拍。我們講三次交握，快速地講：左邊是我們的 Client 端，右邊是 Server。一般來說我要跟 Server 建立連線的話，第一個我要告訴 Server 我要跟你連線，所以我會丟一個 SYN 的 flag 跟它說我要跟你連線。那 Server 收到以後，它要回覆——你要不要，你不愛我也要講，所以你要回它 ACK，ACK 就是好、OK。但是同時 Server 要跟你連，所以它也會傳送一個 SYN 的 flag 跟你說 Server 要跟 Client 端連線。最後 Client 端就要回覆它——你愛我，好那我知道了，但是你也問我那我愛你嗎，我也要回覆它。所以就是要互愛，我們才能建立關係的連結。這個是正常的三次交握的流程。

如果我們這樣建立完一個連線，那我的 Server 就會被佔用一個連線（槽）。那什麼叫做 Connection Flood？就是我有很多台主機、幾千台幾萬台都跟你這台 Server 建立連線，而且這個連線是完全連接的，就等於是你一個人對付很多個女朋友這樣，完全連接。這個是 TCP 的 Connection Flood。

但是啊，如果我們丟了 SYN 過去以後，Server 回傳我 SYN 加 ACK，最後我沒有回覆它「我愛你」、我沒有回覆它 ACK，那這個連線就沒有成功建立。那請問 Server 這個時候會怎樣？它會等，它會等你告訴它「你愛它」，它會一直等，等到海枯石爛、等到它資源沒了、等到錢都花光了。這個就叫 SYN Flood。因為上面那個是 Connection——你要完全建立三次交握以後，然後你就佔用它的（連線）一直佔用，然後你可能都不太丟流量，完全佔用完以後它就沒有連線數了，這個比較辛苦。那比較簡單的就是 SYN Flood，我就一直丟 SYN，然後你回覆我以後我不回你，所以你就會等我，你的資源就會吃在那邊。一般來說 Server 的連線數大概可以到幾萬，但是它不是一開始預設值就可以幾萬的 Connection，還要設定，然後可能會跟你的 Server 的一些資源有關係，你的資源越多能建立的 Connection 是越多的。所以預設像 Nginx，它這個 worker 的 Connection 一般來說預設是 1024，所以就看你建幾個 worker，如果你只有建 4 個 worker，你大概就是 4096 個 Connection。那我只要有 4000 多台、4096 台一直跟你連線，那你就沒有連線數了，那正常使用者就進不來。那請問這兩個攻擊是比較偏哪一層？（聽眾：第四層。）第四層，就是 TCP、UDP 的。

好，那接下來我們再來看，這個叫第六層呈現層。呈現層在做什麼？大概做三件事：加解密、壓縮解壓縮、還有編碼 encode，比如說什麼 Base64、Unicode，大概主要是做這三件事比較多。那我們在第六層呈現層，因為它有做加解密，所以一般來說我們 TLS 的——大家知道，因為我們從傳統的 HTTP 傳輸是沒加密的、是明文的，變成說我要傳輸要加密，所以會有 TLS 憑證。TLS 它是加解密，所以它比較偏向在第六層。它有一個常見的（攻擊）叫做 SSL/TLS Abuse，Abuse 就是虐待它、濫用它。怎樣虐待它？因為 TLS 它要跟你做傳輸加密，它要跟你互動一下，互動完成，我們商量好一個加密的約定——就有點比如說「我以後不要講我愛你，以後講我討厭你就是愛你的意思」，就是要協商我們之間的對話內容，所以我們也要做類似那種連接的感覺。所以 SSL Abuse 是指說：我一直頻繁地跟你去協商說「我們來做傳輸加密、來做傳輸加密」，一直跟你做這件事情，我只要建立完這個 TLS 的傳輸加密的協定以後，然後我就把它中止掉，中止掉以後再跟你建立一次，然後再中止掉、再跟你建立一次，就是一直頻繁地跟你去玩這個「我又進來了、我又出去了」這個遊戲，然後你就會被我煩死。一般來說你要建立這個 TLS 的連線的話它是很吃資源的，因為加密解密是很吃資源的，而且我還要去告訴你說我們要怎麼樣做加密解密、要傳很多資料。所以這個就是 TLS 的 Abuse。

它最著名的攻擊是這一個，叫 TLS Handshake Flood。講這個攻擊，我們要看一下這個流程。我們要先知道說 TLS 有好幾版，那目前已經被棄用了、不能再用了，就是 1.1 版以下、包含 1.1，1.2 還能用、但那也（有）不安全的，現在都建議要用 1.3 版。那 1.2 版它怎麼跑？我們一樣，左邊是 Client、右邊是 Server。所以一開始它們兩（者）先建立好連線，基本上我們前面的 TCP 的三次交握要先完成。我們建立連線以後，我才會跟你協定說「那我們傳輸加密怎麼搞」，所以前面已經做完 TCP 的全連接了。接下來我們就要開始做怎麼做傳輸加密：第一個 Client 會跟 Server 說 Hello、我要跟你傳輸加密，會發一個 Client（Hello），這個時候它會告訴 Server 端說我是用什麼樣的一個密碼套件——因為你要做加密傳輸，那你要告訴 Server 我大概是用什麼密碼套件。然後 Server 這邊確認一下，它會說 Hello、那我們這次用某一組（密碼套件），譬如說用 AES。這個時候它還會給你一份 certificate，這個是什麼 certificate？就是 X.509 的 certificate，就是我們會有一個信任鏈、會有一些 CA 的機制，要確認說現在是合法的、你是合法的，那我跟你連接是正確的，所以要給你一個 certificate 去確認一下。同一時間 Server 也會給你——我們要交換金鑰，因為我自己本身會有私鑰去加密，那你要解密怎麼辦？我們講非對稱式的加解密，我用我的私鑰、你要用我的公鑰去解密，所以我就要告訴你說好那我們用這個密碼套件、那我現在給你我的公鑰。那我要給你公鑰，會有一些公鑰交換的機制，那常聽到譬如說 DH 的金鑰交換等等。所以 Server 就會告訴你這些資訊。你會發現它一次給的不是一個資訊——不像 TCP 的 Connection 它是 SYN、一個 SYN 而已，然後回來就是 ACK、SYN，很少。但是它這個過去回來是一堆資訊，你看它回來 inline 就給這麼多。接下來我收到了 Server 這些的資訊以後，我是不是要回它，跟它說好那你給我的這個交換的 key 以後，那我們也要來交換一下我的 key。然後最後我傳送——我們都已經交換好 key 了、我們都可以互相加解密了，那我就傳一個 finish，這個 finish 是我加密過的訊息，我已經加密了。所以到紅色這邊，我現在傳輸就已經加密了，如果 Server 能夠解密那就結束了，接下來所有傳輸的資訊都是加密，因為我們已經協商好了。這個就是 TLS 1.2。那我們這樣子來回兩次，過去回來一次、過去回來一次，來回兩次，所以它叫兩 round、它叫 2-RTT。

那為什麼會有 1.3？因為 1.2 還有很多不安全的密碼套件，比如說 RC4、SHA1，還有像是 3DES，它還有很多舊的密碼套件。如果你們的舊網站有用舊的密碼套件，那很容易被破解，所以它有 1.3 的出現。那 1.3 除了去移除很多舊的密碼套件之外，它還有做一些事情。我們來看一下 1.3 怎麼玩：因為它移除了大部分不安全的密碼套件，所以它沒剩幾個、好像只剩三四種，而且常用的就那一兩種。所以 Client 端它可以猜——Server 應該是用這個密碼套件，直接猜說「那你應該是用這個密碼套件喔」，因為大家都用這個。那因為我已經猜中了、假設我自己猜中了，我這個時候就可以給你我們共用的 key 的一些資訊、我現在就給你。因為在前一回合（1.2 的時候），它是 Server 告訴我我才給你；但是在 1.3 的時候，因為你剩下的密碼套件太少了，我一猜就猜中了，所以我在這個時候就直接給你相關我們要做金鑰的一些（資訊），譬如說橢圓曲線演算法的一些資訊給你。那 Server 收到了這個資訊以後，它馬上就可以跟你說這個密碼套件我們有、而且我也可以用這個，接下來它也會跟我做金鑰交換，然後它要做橢圓曲線的一個計算，所以它也會給我一個 key share。接下來它還是會給我們相關的譬如說 X.509 的這個資訊，然後以及它會做數位簽章，然後最後傳輸最後一個 finish。基本上我們在第二回合丟 X.509 的時候就開始加密了，這邊就開始加密了、跟前面不一樣。前面（1.2）是 X.509 我是沒有加密的狀態傳給你，但是在這個時候我已經加密傳給你，那是不是比較安全？而且它們用的密碼套件都是最安全的、都是經過驗證的，不安全的都刪掉。所以這個時候我們 finish 已經加密了，如果 Client 端確實可以解密，接下來我們所有的傳輸都是加密。所以它只要跑一 round 就結束了，甚至它可以跑零 round——怎麼跑零 round？就是我們之前連線過了，那我就不用再跑一次，這是零 round。那這個有什麼好處？除了比較安全之外，它也會比較快，因為你跑兩 round 跟跑一 round 就差一倍了，那速度當然會快一點。一般來說你這樣子跑過去 50 毫秒、回來一次 50 毫秒，那你少了（一個）round 是不是就少了 100 到 200 毫秒，就會比較快、而且比較安全。所以一般來說你們用 1.3 它的速度基本上會比較快。那建議都換 1.3，除非你們有一些比較舊的系統改不動了、都是 1.1、1.2，那就無解，那就要有其他的補償措施，譬如說你們要放在內網、網路區隔，還是裝很多的什麼防火牆等等。

那我們這個 TLS 它交換資訊很複雜，如果我一直頻繁地跟你交換、然後又把你中斷，又跟你交換成功、又中斷，那你是不是你的資源就被我吃光？這個就是我們前面提到的 TLS Handshake Flood，就是我用洪水的方式讓你一直頻繁地在做這個 TLS 的交換。這是第六層的攻擊。

接下來我們來講第七層。第七層的攻擊大家比較常見，譬如說 HTTP 的 Flood，就是一直發送很多的 HTTP request，就是我們常講的那個殭屍電腦——操縱幾百台幾千台殭屍電腦，然後一直用那個 HTTP 的 request 去把你的網路流量或 CPU 充滿，這個就是最常見的。那還有其他東西，譬如說這個叫做快取的 Busting。一般來說大家有用過 CDN 嗎？內容交付網路，就是把你的圖片、影片跟一些靜態的網站 HTML、CSS、JS，放在離客戶比較近的邊緣節點，這是 CDN 的一個機制，比較常見的就是 Cloudflare 跟 Akamai，然後各大雲都有自己的 CDN。因為它要快取，工程師有時候換了一些圖片影片或是 HTML 的文字，他想要趕快看到結果，但是因為它已經被快取過了，他看到的就（是舊）資訊，這個時候他會加上一些參數，譬如說問號後面都會加參數——它原本是 v1.1，但他把那個 v 改成 1.2、就改個參數，那他就可以觸發重新快取。這個機制看起來不錯，可是如果是惡意人士來用，就是你的快取永遠都沒有用。因為快取的目的就是希望讓近的客戶可以趕快看到東西，然後甚至我們 Server 不會一直被很多的大流量進來請求這些靜態檔案；但是如果他攻擊，他會跑一個機器人，然後它就一直幫你跑 v1.1、v1.2、v1.3、v1.4，一直用不同的參數去查詢的話，那你原來的快取都是失效的、沒有用，你的 Server 就會一直被查詢這些靜態檔案。這個叫做快取爆破。

再來下一個叫 DNS Query 的 Flood。一般來說我們會去 DNS Server 去查詢一些資訊，DNS Query 就是你要把這個網址轉成 IP，所以它裡面有一些資訊、它會去轉換、去查詢。DNS 的查詢它是很吃資源的，它吃 CPU、吃資源。所以我們就大量地發送查詢到 DNS Server 去查詢，讓它負載變高。而且我現在的查詢不是正常的查詢，我查的是一些沒有的網址、沒有的網址之類的，就查一些沒有的，那你這個 Server 可能會疲於奔命，因為你沒有記錄過，你可能還要去做一些後續的處理。這個就是 DNS Query 的 Flood，大量查詢不存在的或是隨機產生的這些子網域，它就會吃很多的 CPU。那我問一下，這個 DNS Query，雖然我把它列在第七層，但是一般來說我們做 DNS Query 是用什麼協定？（聽眾：UDP。）UDP、53，所以有時候它也是屬於第四層。但是因為 DNS 一般來說我們是在應用（層）為主，所以它可以列在第七層，但有時候它也可以列在第四層。所以為什麼有一些我會把它混在一起，因為它到底是第三層、第四層有時候有點難分清楚，連那些大公司都把它混在一起。好，這是簡單跟大家分享。那等一下我會講 DNS 在架構面怎麼樣去處理這些問題。

這裡先講一些其他案例。譬如說 Fortinet 它有一個 report 說，有很多的智能牙刷被木馬感染——智能牙刷，然後有三百萬支被感染，然後它們可以受駭客的遙控去攻擊某些網站。三百萬支牙刷，我想說（怎麼）牙刷（也）連網路——牙刷連網路更（詭異），然後還被駭客入侵。這些牙刷是屬於這種比較偏 IoT 裝置，IoT 裝置最大的問題就是它很少（更新）修補，幾乎一出來以後三個月內就會有一個 CVE，就很容易鑽進去。譬如說我之前有看過什麼 AI 馬桶蓋，那馬桶蓋會去監控你出來的便便，就是（看）是不是正常的形狀顏色，來看你是不是有生病，那個都很危險，我常常講這個案例。一般來說（用）殭屍電腦（做）DDoS 攻擊，那些殭屍電腦幾百台幾千台就很（厲害）了，但是這三百萬支，所以（駭客）有一個超強的軍團。

這一個案例，這幾年大家有聽過，雲端廠商他們有一些物件式儲存，例如說 AWS 的 S3、Azure 的 Blob，然後 GCP 它有自己的類似物件儲存。物件式儲存最大的好處，第一個它永遠不會壞掉、幾乎是一萬年才會壞掉一次，就是我們有生之年都不會看到那個檔案壞掉；第二個是你有多少流量我都吃得下來，非常高可用，它沒有網路流量的限制，它非常強。所以曾經就有人放了一個圖片在這個物件式儲存裡面，然後這個圖片被 DDoS 攻擊。那大家知道雲端吃的那個費用又很大——一塊是流量，你只要檔案從雲端離開到 Internet 就會收費。所以（有人）對這個檔案 DDoS 攻擊，它每秒 500Mbps，DDoS 兩天吃了一萬一（美元）的（帳單），這樣攻擊很（花錢）。這個 AWS、Azure、GCP 都有（人）吃過這樣的攻擊。那你中了怎麼辦？就是付錢，中了就付錢。那請問這個攻擊怎麼防？因為檔案放在那邊，你就是要給人家看，然後它又沒有流量限制，它無敵，怎麼防？如果 Google 的話，有一個 Google 的 Armor、就是 Google 盾牌，好像可以擋；那如果是 AWS 的話，它的 S3 本身沒有擋、沒有機制可以處理，唯一解法掛 CDN。就是你的檔案不應該是（讓人）直接可以看到原始的檔案的物件儲存空間，它應該是前面擋一層 CDN，然後 CDN 就會在外層先吸收這些流量，然後 CDN 它本身也會有一些惡意攻擊機制，它會去擋。所以像 AWS 的話，你可以掛它的 CDN，或者是掛 Cloudflare，都可以去解決這個問題，因為 CDN 的機制本身就會去擋。這個是一個案例。

再來，有很多雲端主機都變成 DDoS 的跳板。雲端主機最大的好處是它的網路流量幾乎是無限的，所以它網路流量是很難被吃滿的。我們公司網路流量頂多 100Mbps，很快就吃滿了；雲端他們的網路流量都是以 G 起跳的，幾 G、幾十 G，那他們這個流量就很適合拿來做 DDoS 的攻擊機。所以這個跟大家分享一下。

那最後壓軸就是這個。那針對這麼多的 DDoS 攻擊，到底怎麼防禦？沒有辦法用一個機制擋，這個是第一個要知道的，沒有辦法用一個機制擋，所以它（要）用縱深防禦。那怎麼樣的縱深防禦法？我們下面這個是正常的使用者 User，正常 User 一般看網站就是用 HTTPS。那有一些譬如說工控，或者是 Server 對 Server 的一些（無需）協定，它可能不一定是用 HTTPS、可能是用 TCP 或 UDP 的（協定），或者是像一些工控的他們可能比較多用 MQTT，就有好幾種模式，這個是正常流量。我們看 DDoS 流量上面的這個：DDoS 流量的話，AWS 第一關，它在全球的 infra（基礎設施）——我們要先知道一下，它會有全球的 DNS 的功能跟全球的 CDN 功能，它們在全球的這些 infra 裡面，有一些是自己的機房、有一些是跟當地的 ISP 合作，譬如說中華電信、遠傳這種 ISP 跟他們合作用他們的機器。所以它全球都會有一個盾牌（Shield）的服務，它可以預設幫你擋第三層、第四層的 DDoS 攻擊或是 DoS 攻擊，這是它的第一道關卡。

如果攻擊者能夠通過這個第一道關卡的話，接下來下一道要進到 DNS。DNS 如果你是自己建一台 Server 的話就一台；如果你是用雲端業者的 DNS 服務，它不是一台，它是幾萬台。所以我們在這個 Route 53，這個 AWS 的 DNS 服務，它有一個功能叫 Anycast。Anycast 是什麼？就是一般來說我們會去 Ping 8.8.8.8 一個 IP，理論上一個 IP 是對一台 Server；但是它的 Anycast 可以讓你這個 8.8.8.8 後面是一批 Server，一批 Server 全部都是用這個 IP。所以它全球有一堆的 DNS Server。它今天在 Ping——因為我不知道 AWS DNS 是哪個，我們用 8.8.8.8 比較有名——如果它處理的是 8.8.8.8 之類的，它要去做 DNS 的 Query Flood，就是你要做大量一直查詢、一直瘋狂查詢讓它的 CPU 吃滿，是有點難的，因為它有幾萬台。你要讓一台吃滿效能是很簡單的，但是你要讓幾萬台吃滿效能，那你可能要有幾十萬台去攻擊它才有可能，但是你要有幾十萬台的攻擊機有困難。然後包括說這個 DNS 的反射攻擊，它在反射的時候不會馬上反射到你的主機，它還是在他們那一群的 DNS Server、那些節點在反射，所以它會先幫你吸收掉，然後它發現異常它會先擋掉。所以在第七層的這個攻擊，這個地方它就會擋掉 DNS 強灌的攻擊。

再來我們經過 DNS 以後再往下，理論上它會解析為一個 IP。但是在進去你的 Server IP 之前，有一些靜態的圖片影片我們可以用 CDN 的機制去把檔案讓比較近的客戶先看到。這些 CDN 的 Server 可能是有幾萬台幾十萬台，它們也可以幫你吸收流量。而且那些圖片影片，或者是我們玩遊戲的這些壓縮檔幾 G，都不是從你 Server 來的——我的意思是說，它不是去你 Server 要這個流量、要這個檔案，它是在邊緣節點跟你要這些檔案這些流量，所以你的這些大量的流量在邊緣就會被 AWS 吸收掉，它就會幫你吸收這些大量流量，不會一瞬間你的客戶就會被充爆。這個是 CDN 的部分。所以它可以解決什麼？譬如說我們前面提到的 HTTP Flood，就是它有大量的 request，以及 TLS 的 Handshake，其實在這邊它也會先解決掉。以及這個叫 Edge Cache，就是它的快取可以幫助我們把流量給緩解，而不會全部衝到 Server、Server 就掛了。

我們在這個 CDN 的地方還可以去掛上 WAF。WAF 是網站應用程式防火牆，它是掛載在第幾層？（聽眾：第七層。）大家都知道。所以第七層常見的除了 HTTP Flood 之外，還有一些——剛剛我有網站被攻擊的那個案例，就是每幾個小時被 request 幾萬次，它本身可以去設定這個叫 Rate-based 規則，就是幾分鐘它請求幾次就會自動封鎖。但是它的封鎖是暫時封鎖、不是永久封鎖，所以還是要自己加一個自動化黑名單它才會永久封鎖，不然它是暫時封鎖。那我們第七層還有很多的阻斷東西，譬如說 SQL Injection，把你的一些資料竊取，甚至可以取得你的管理者的權限，然後讓你的服務給中斷；還有一些什麼 XSS，或者是 AI 會去爬蟲、會去各網站查，它（WAF）也可以幫你去阻擋這些 AI 的爬蟲，還有一些機器人的攻擊，這個在 WAF 的地方可以解決。

那我們這邊如果擋完以後還是有惡意流量進去怎麼辦？它就會繼續往裡面。然後我們在 CDN 它會放靜態的檔案。我們前面是不是有提到 GCP 的攻擊案例，就是雲端的物件式儲存被 DDoS 攻擊，所以它唯一的解法就是 CDN。那 AWS 也有自己的物件式儲存叫 S3，它本身可以把你原本 Server 的靜態圖片跟影片就放在這邊，它不會影響到你 Server 的一些效能。所以你等於就是把你的一些原始的資料放在其他地方，然後用 AWS 的其他服務去分散 Server 的負擔。所以你放 S3，然後再搭配 CDN，就可以防禦那種單個圖片 DDoS 攻擊、讓你的帳單爆表的這個現象。

然後最後我們終於進來了我們 Server 的網段。進來以後，我們這裡有一道——你要進來這個網段之前，我有一個 Gateway，這個叫 Internet Gateway，它是一個網際網路的大門，你要進到 Server 之前要先統一進這個網路的大門。進大門可以做什麼？我們可以統一做流量控管跟紀錄，我能夠紀錄你的網路流量，如果有異常的話我可以觸發後面的自動化的執行，可以去做其他的阻斷。再往裡面走，我們會到負載平衡。負載平衡在做什麼？我們只有一台 Server 的話很容易被（打）垮，那我至少要換兩台以上。我有兩台以上的話，我必須要有一個機制在前面去偵測說我後面有幾台 Server、哪幾台是健康的、哪幾台的流量負載是低的，所以我才把新的客戶的請求進到負載低的那一台、而且是健康的，所以這個是負載平衡器的作用。再來我們是不是會做傳輸加密？那有加密就有解密，我們是不是會掛載一個 TLS 憑證？你要有憑證才能解密，解密是很吃資源的，所以我不要在 Server 上解密，我就直接在負載平衡器去做 TLS 的卸載、我在這邊就直接解密，我的 Server 就吃不到這個解密的資源。那這邊卸載了很多的流量以後，最後我要進到 Server 之前，我還有兩道防火牆：第一道無狀態防火牆，第二道有狀態的防火牆（安全群組）。我們有允許的流量才允許進來、有允許的 IP 才允許把流量進來，所以這邊又有兩道防火牆擋，最後才會進到你的 Server。

AWS Server 的服務叫 EC2。但是如果你的 Server 只有兩台，還是很容易被（打）到，所以它有一個機制叫 Auto Scaling，它可以去判斷你的網路流量的數量、或是你的 CPU 負載，來去決定要不要自動幫你長第三台、第四台、第五台的 Server，來去分散這些 loading，所以它就提高你整個雲端的韌性。經過了這麼多道——幾道？一、二、三、四、五、六、七、八、九、十道。如果最後你真的通過了十道關卡進到我們的 Server，那恭喜你可以 DDoS，但你剩下的量真的很少，然後也許會被我的一些自動化機制給檢測到。那我的自動化機制就在右上角這裡有一個 CloudWatch，它可以去檢視雲端所有資源的一些指標，比如說 CPU 負載、網路流量，還有有沒有一些 log、異常。有異常的話它可以觸發一個事件、自動化執行一些行為，比如說我把（IP 加）黑名單、還是我把它阻斷，我可以寫一些程式去自動化執行，或者是說我發送一封 Email 或簡訊給我們的管理者說「出事了、趕快來看一下」，這個就是附加的第十一道——就是當老是沒有辦法擋住的時候，還是要有人為介入，或者是你已經有設好一些自動化（處理）。所以這是第十一道，要經過十一道關卡進來，有沒有很難？很難啦。

所以新的想法，倒不是說我們要從語意的部分來 block 每一個 prompt injection 可能的組合，而是如何從架構上就讓 prompt injection 這樣子的攻擊可以造成的傷害降到最低。首先你要限制它的能力，那這一部分限制能力其實不是一個很新的概念，對吧？這個放在以前的概念叫什麼？以前一般資安的概念叫 least privilege（最小權限），就是你不要給同樣的一個工具、同樣的一個 agent session 超過它所需要可以做的事情的權限。那 identity（身份）綁定授權，這個也是類似的概念。更（早）像之前十年前五年前你以前在（討論）什麼概念呢？zero trust（零信任），對不對？就是你綁定一個特定的 identity，然後每一次去驗證它，確認它就是它宣稱的那個身份。runtime detection（執行期偵測），就是在 AI 系統正在運行的時候呢隨時觀察它的 output——它的 output 可能是文字輸出，可能是圖形的輸出，可能是系統的動作——如果仔細觀察，然後先定義好說這是一個讀 email 的 agent，那它被賦予的功能就是去讀你的 email、然後理解、然後輸出。如果這個 agent 突然它提出了說「我要讀你的 Google Drive」，這就不對了，所以這個所謂的行動如果發現跟原本的意圖不符合的話，那這是要警示的部分。最後就是 recovery（復原）的部分：當然就是如果發現已經有入侵或者攻擊在發生的話，session 或者是它的 key 是可以去把它 revoke（撤銷）的，讓這個攻擊能夠標記（並阻止）。

第二個概念是所謂的致命三角(lethal triad)，就是 rule of two。其實很多企業在部署 agentic AI 都覺得，那個代理式 AI 好像會做很多事情，我不知道它實際上會做什麼，感覺很不安全。其實站在工作流的角度，它可以是非常好用的一個工具，只是說我們普遍會認為說，我不知道它在做什麼。那這三點：輸入、還有接觸到的資料跟系統、還有狀態改變，這三點呢其實就是所謂的 lethal triad 這個三角。那同一個 agent session 如果同時持有三者的話，那其實就非常可怕，它就可能可以做很多——可能會造成潛在破壞性很大的事情。所以我們在設計系統的時候，盡量做到責任的分層、盡量做到責任的區隔，這個一方面是做好系統的權限管理，二方面是如果真的哪一個 session、哪一個 agent 一旦被侵入、被 compromise 的話，它所造成的傷害是可以減到比較低的。

第三個安全趨勢是：MCP 是新的信任邊界(trust boundary)。這個 MCP 協定出來之後，各位——各位的組織或企業有在用 MCP 的舉個手看一下？OK OK，可能兩三成就在用 MCP 的服務這樣。那我自己接觸到的一些潛在的客戶或者是朋友，他們會覺得說好像不太敢用 MCP 的原因是因為，有的時候我並不知道它按（下去）安不安全——一個 MCP server 它架在那邊，好，那我的 LLM 決定去 call 那個 MCP 的服務了，很好、很方便，可是那個服務或者提供那個服務的公司，是我可以信任的嗎？真的是我可以相信的服務嗎？會不會有任何惡意的程式或者是惡意的指令藏在裡面呢？我不知道，對吧？前一場——在另外一場，Henry（大會共同創辦人）的演講也講到說，第一個 malicious MCP server，對吧？就是——他們前 15 版都是非常正常、沒有問題的，第 16 個 release 就藏了一些 malicious code。所以 MCP 的價值是讓 AI 的能力更加地延伸、也更加擴充，但這也正是 MCP 的資安風險所在。所以在考慮設計 AI 資安的 program 的時候呢，我們絕對要考慮延伸出去的這一部分。

第四個趨勢是：Agentic IAM（代理式身份與存取管理），就是 Agentic 的 identity and access management，它會成為企業 AI 核心的控制點。那各位可以看到，商量一點——誰授權、何種身份、權限最小化、註銷輪替，還有留存證據，這個其實也是以前的 least privilege，我們在 IAM 的原則，其實在 AI（時代）是沒有改變的。那只是說，因為在 agentic AI 正在風行的時候，以前我們有非常多的事情是由人來做，現在是由 agent 來做，所以我們需要賦予它身份、賦予它身份以下的權限。那但是（有時候），一方面呢，這個權限有時候給的過寬或者給的不恰當，那造成了可能潛在會造成的系統傷害，那就得不償失。所以 Agentic IAM 的管理，其實會是下一個在 AI 資安當中很重要、需要注意的事情。

那最後一個我想要談的是：Memory 跟 RAG，讓這個知識庫其實會變成一個新的攻擊面。那這個也很容易理解，大家在這邊如果這兩三年熟悉資安，然後又熟悉 AI 的話，應該聽過 data poisoning，就是資料投毒。好，今天如果我的 LLM 系統我想要加強它的知識，那在它裡面設立出這個 RAG，就是它可以去截取外面的資料來加強它的知識，那外面資料如果裡面是有惡意的指令，或者是說有惡意的資料或者是錯誤的資料的話，就會造成整個危機的擴大。所以呢，數據投毒，然後記憶潛伏，還有來源追溯斷層，這些都是 Memory 跟 RAG 會讓知識庫變成新型攻擊面的原因。

那我這邊特別提出這個東西的原因是因為，這個東西經常被大家忽略，大家知道講——我的 RAG 讀了一些東西，那它就存到它的長期記憶裡面去了，也許這東西是不大好的，或者是錯誤的，或者是惡意的，可是其實如果我不去動這個東西的話、沒有人問這個相關東西的話，我一輩子也不會發現。所以類似這樣子的攻擊面呢，是比較潛藏的，所以我想要特意在今天這個機會跟大家提出來說，如果您在部署 AI 系統的時候，這一塊就是 Memory 跟 RAG 的安全性要特別注意，因為它有可能會變成一個中長期的隱患。你可能會覺得我的 AI 資安都已經做得很好了，為什麼欸久久突然有什麼事情就會出錯、什麼什麼 Tool 又會被一段？為什麼？很可能就是出在這個記憶方面的部分。

那這一個是我認為比較實用的 AI risk（AI 風險）的思考方式，那當然這是指著我自己的想法，因為分析風險有各種不同——有 FAIR，有各種不同的方法論，所以這是我自己的看法，各位參考一下就好。就是 AI 的風險來自於：它能夠做多複雜的事、能力有多大，然後呢被授權了多少，以及它本身可以接觸到多少資料還有工具、還有工具流，以及最後如果企業對這些都沒有——都看不到、都沒有好的（掌握）、都不確定我的 AI 系統做了什麼的話，那這整個 AI 的風險就是高的。

好，那因為時間關係，所以我快速帶過三個 AI 在企業中的常見應用場景：Coding Agent、然後 Internal RAG 的助理、還有 Finance Agent。那 AI Agent 的話大家知道，現在 input 可能是外部不信任的來源，那 agent 在自主規劃跟執行的時候也有可能出錯，那 tools（存取工具）的時候也有可能會選擇到過度擴權 或者是不恰當工具，然後最後它會造成狀態的變更，會直接影響到你們的 production（生產）環境。那如果只講一句話的話，核心原則就是：我們會允許 AI Agent 自主產生各種的修（正）還有 patch，但是呢我們不要讓它繞過審查人員、還有 CI/CD pipeline，或者是我們已經部署好的防線，這是最低的底線。

那 RAG 助手的話呢，我們知道 RAG 剛剛講的有可能會受到（data）poisoning 這種攻擊的影響，那所以這邊的關鍵問題是：如果一個惡意的、被污染的知識片進入回覆的話，它會不會觸發下游的行動？比如說剛剛講的執行一些它不應該執行的事情？如果會的話，它就不是單純的 RAG（檢索增強生成），在各種不同的節點我們就需要去做恰當的控管還有限制。

Financial operations 跟 Support agent 的話，其實一般的 agent 也沒有什麼特別的故事可以講，但是碰到 Financial 這個事情，對不對，那是不能出錯的。所以這個例子要講的就是說，有些可能在技術上角度我們不覺得特別嚴重、或者特別複雜的事情，它在商業上的影響可能是很大的，純粹是因為它的用法不一樣，所以這個也是要注意的部分。

那我想要快速帶一下，就是如何用一個開源的 AI Defense 這個 framework 來當作 AI 防禦的共同語言。那 AI Defense 是一個把 AI 的風險轉換成防禦手法的框架，那我這邊快速把它打開來——這個都可以在（官網）找到。那這個框架呢，主要是在不同的 tactics、不同的戰術底下：model harden、detect、isolate、deceive、devalue、restore，底下呢有不同的實作手法。那以這樣的一個框架來講呢，我舉個例子來講——例如說 adversarial robustness training（對抗式強化訓練），我們會需要對 AI model 做敵對式的強化訓練，那這一個防禦手法點進去呢，這個框架會告訴你說有哪些 implementation guidance（實作指引），實際上要做哪些事情，那實際上做的哪些事情下面呢又有該做的事情，還有一些 example code。那我知道 2026 年現在大家都不會看，所以我們要做的事情就是把這個東西怎麼樣 copy 下來，或者是把它擷取下來，然後衛星 AI，看看怎麼樣符合自己公司裡面系統的部署。那如果有興趣的話呢，也可以參考同一個 robustness training 底下的開源工具，或者是如果預算夠，或者是這一個部分對您的企業、對您所在的組織是很重要的話，您也可以考慮一些商業的軟體，這樣子。

那這個框架還有一個不錯的部分，說它會對應到目前很多流行的 AI 資安框架，例如說 MITRE ATLAS，這個是由 MITRE 所提供的 AI 系統攻擊的一個 technique（技術分類），那它針對每一個攻擊的步驟——這個 kill chain 底下呢，它都有詳細介紹是什麼。那這個框架底下它會針對每一個（技術）的詳細介紹呢，你點開來，針對這一個 Exploit Public-Facing Application是針對這個攻擊手法呢，總共有這一些可能在 AI Defense 裡面可以防禦到這個攻擊手法的方法。所以這是一個如果在實作上是可以很好參考，來如何在自己的系統面做好 threat modeling（威脅建模）之後，可以參考 AI Defense 的內容來看怎麼樣做資安的保護還有防護。

那另外一個是從落地到真實——就是從真實威脅到可落地的方法，就是 AI Defense Labs 裡面的 AIA 還有 Secure Flow，那這是我小弟(自謙語)設立的另外一個網站叫 ai-defense-labs.com。那這個點進去呢，就是它會在防禦實證底下，你可以看到過去一段時間在真實世界所發生的 AI 事件，好，那這些列出來都是真實世界發生的 AI 事件。那例如說 shadow AI 的問題(影子 AI 的問題)，那這一篇分析了不同報告，發現了不同企業裡面影響程度大小不一的 shadow AI 問題，然後它有一個給大家可以讀的威脅分析，就是 shadow AI 是什麼樣子的一個威脅、怎麼發生的，然後再下面就是如果貴組織或者是你有興趣想要增加自己對 shadow AI 防禦的知識的話，有哪些潛在的 AI Defense 的防禦手法是可以防止 shadow AI、或者是減緩 shadow AI 這樣子狀況發生的，這一個對應這樣子。那裡面每一個目前都是有完整對應的一個方法。

好，那 Secure Flow 是另外一個基於 MITRE ATT&CK Flow 的東西，那它會把一些過往 AI 真實的 incident（真實事件）把它化成每一步——這個攻擊是怎麼發生的。好，那例如這個 OpenClaw command and control，大家可以看到它的攻擊步驟就是從 search open website 開始，然後到 acquire public AI artifact 等等，那它有分成下面好幾步，這些都是實際過去攻擊的真正軌跡這樣子。那這是 MITRE 開放給大家、大家都可以看到、大家都可以用的。那在 AI Defense Labs 這個網站上面呢，我們做的是就是說，針對每一步這個攻擊、每一步它的攻擊步伐底下有什麼樣的 AI Defense 防禦手法是——如果你已經部署這個防禦手法，就可以比較可以防禦或者說減緩這樣子攻擊的一個狀況。

然後最後我想要快速分享一下，我覺得在未來的一段時間——半年到一年，如果你要注意相關的 AI 技術框架或者是知識的話，那這一張圖的含金量我自己覺得是很高。首先第一個，AI（相關的）OWASP 這個 AISVS，這是一個蠻久（已經）存在很久的、針對應用程式的安全去確認的一個標準，那 AI 系統這幾年出來了，所以大概是——這幾乎是上個禮拜、這一兩個禮拜的事情，他們剛推出了新版的、第一版的正式版 AISVS（AI Security Verification Standard），裡面有很多的資訊可以讓你在部署你的 AI 系統的時候好好地 check 我有沒有做到這個確認標準裡面應該叫我應該要做到的事情。那 OWASP 另外還有很不錯的一些 publication，因為 Top 10 也是非常值得參考的。那如果你對攻擊方比較有興趣的話，NIST 的 Adversarial Machine Learning（AML），就是 NIST AI 100-2，它是一個還不錯的攻擊分類的文件，大家也可以去參考。那 MITRE ATLAS 剛剛提到了，這是一個由 MITRE 提供的 AI 攻擊的完整框架。然後 ISO 42001，如果做 AI 大家都了解吧，這是一個 AI 管理的框架。那我想啊，這邊要特別提的是 ISO 27090，這個是 AI cybersecurity 的一個 guidance，那不用去找，它還沒有出、它現在還沒有 publish，預計應該未來幾個月或半年左右會 publish 這樣子，那這是一個很值得注意的 guidance，我自己也非常期待這樣。

NIST SP 800-53，本身是 NIST 美國國家標準技術局的 SP 800-53，目前是第五版，它是一個針對資訊系統的安全還有隱私要（做）控管的一個完整名單。那它現在正在推一個東西，也是一樣還沒有完全做好、還沒有出——就是安全跟隱私控管呢，如何應用到 AI 系統上面的一個狀態，所以這一個文件出的時候大家也可以注意一下，我覺得應該是對於做好 AI（資安）會非常有幫助的。然後最後 AITBM 就是待會今天最後一場 Henry 會介紹的，是我目前認為在業界或者是實作上很可能是最前沿、最符合實際狀況的，對整體 AI 系統有一個風險評分（risk scoring）的一個方式。那 AI Defend 就是剛剛提到的，我自己、還有社群所做出來的這個框架也是可以參考的一個東西。
